# 修复项目问题

Created At: Jan 9, 2026

当识别到用户说"修复："或"修复问题："时，执行以下步骤：

## 1. 解析问题内容

提取"修复："或"修复问题："后面的内容作为问题描述。

## 2. 确认不明确的问题

如果问题描述中有不明确或模糊的地方，**必须先与用户确认**，再开始规划。常见需要确认的问题：
- 问题的具体表现和复现步骤
- 期望的正确行为
- 问题的影响范围
- 是否有相关的错误日志或截图

## 3. 分析当前项目现状

- 定位问题相关的代码模块
- 了解相关功能的实现逻辑
- 排查可能的问题根因

## 4. 制定修复计划

仔细思考，认真分析，规划内容应包括：

### 4.1 问题概述
简要描述问题的核心表现和影响

### 4.2 根因分析
分析问题产生的根本原因

### 4.3 复杂度评估
预测修复复杂程度（低/中/高），说明评估依据

### 4.4 分阶段修复计划
将修复任务分成不同阶段，每个阶段包括：
- 阶段目标
- 具体修复任务
- 预期产出

### 4.5 风险与注意事项
列出修复过程中可能的风险和需要注意的问题

## 5. 保存计划文件

将修复计划文档保存到 `docs/` 目录下，文件名格式：

```
yyMMdd_HHmm_修复_问题说明.md
```

示例：`261204_1620_修复_控制失败.md`

## 注意事项

- 修复前必须确认所有不明确的问题
- 优先定位根因，避免只修复表象
- 复杂度评估要有依据
- 考虑修复对其他功能的影响
- 建议增加相关测试用例防止回归
